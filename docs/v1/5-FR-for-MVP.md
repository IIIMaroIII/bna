# Functional Requirements (FR) для MVP

## Чёткий, однозначный список требований

Ниже — **формальные функциональные требования**, без реализации, технологий и архитектуры.  
Это то, по чему **можно начинать разработку** и **рисовать дизайн**.

---

# Скорректированные Functional Requirements (FR) для MVP

## 1. Public Presentation Layer

- FR-1: Система должна предоставлять публичную Home Page с описанием продукта.
- FR-2: Система должна предоставлять страницу Features с описанием возможностей.
- FR-3: Система должна предоставлять страницу Pricing с визуальным отображением тарифов.
- FR-4: Система должна предоставлять страницу Contacts.
- FR-5: Система должна предоставлять страницу Feedback с формой обратной связи.
- FR-6: Public pages должны быть доступны без авторизации.

---

## 2. Общие требования системы (App Area)

- FR-7: Система должна поддерживать работу нескольких компаний с полной изоляцией данных.
- FR-8: Все пользователи системы должны принадлежать ровно одной компании.
- FR-9: Пользователь должен иметь доступ только к данным своей компании.
- FR-10: Система должна поддерживать роли пользователей: Owner, Admin, Employee.

---

## 3. Регистрация компании и Owner

- FR-11: Система должна позволять зарегистрировать новую компанию.
- FR-12: При регистрации компании система должна автоматически создавать пользователя с ролью Owner.
- FR-13: Owner должен иметь возможность входить в систему по email и паролю.

---

## 4. Аутентификация и авторизация

- FR-14: Система должна позволять пользователю входить в систему по логину/email и паролю.
- FR-15: Система должна определять роль пользователя после авторизации.
- FR-16: Система должна ограничивать доступ к функциям в зависимости от роли пользователя.

---

## 5. Управление сотрудниками (Admin / Owner)

- FR-17: Admin и Owner должны иметь возможность создавать пользователей с ролью Employee.
- FR-18: При создании сотрудника система должна назначать ему credentials.
- FR-19: Admin и Owner должны иметь возможность деактивировать сотрудника.
- FR-20: Деактивированный сотрудник не должен иметь доступ к системе.

---

## 6. Управление проектами (Admin / Owner)

- FR-21: Admin и Owner должны иметь возможность создавать проекты.
- FR-22: Проект должен содержать как минимум название и статус (active / inactive).
- FR-23: Admin и Owner должны иметь возможность назначать сотрудников на проекты.
- FR-24: Employee должен видеть только назначенные ему проекты.
- FR-25: Система должна запрещать выбор неактивного проекта.

---

## 7. Учёт рабочего времени (Employee)

### Clock-in

- FR-26: Employee должен иметь возможность выполнить clock-in.
- FR-27: При clock-in Employee должен выбрать проект.
- FR-28: Система должна запрещать clock-in при активной сессии.
- FR-29: Система должна проверять подключение к рабочему Wi-Fi.
- FR-30: Система должна запрещать clock-in без разрешённого Wi-Fi.
- FR-31: Система должна сохранять время начала работы.

### Clock-out

- FR-32: Employee должен иметь возможность выполнить clock-out.
- FR-33: Система должна запрещать clock-out без активной сессии.
- FR-34: Система должна сохранять время окончания работы.
- FR-35: Рабочая сессия должна завершаться после clock-out.

---

## 8. Просмотр рабочего времени

### Employee

- FR-36: Employee должен иметь доступ к своим рабочим часам.
- FR-37: Employee должен видеть часы по датам и проектам.

### Admin / Owner

- FR-38: Admin и Owner должны видеть рабочее время всех сотрудников.
- FR-39: Admin и Owner должны видеть рабочее время по проектам.
- FR-40: Система должна отображать активных сотрудников (clocked-in).

---

## 9. Бизнес-правила

- FR-41: У сотрудника может быть только одна активная рабочая сессия.
- FR-42: Рабочая сессия должна быть связана с пользователем и проектом.
- FR-43: Все рабочие сессии должны сохраняться для просмотра.
